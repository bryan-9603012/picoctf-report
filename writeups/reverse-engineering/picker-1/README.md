## 二、題目概述

本題提供一個選單式的互動程式，使用者可以透過輸入選項來執行不同功能，例如：

- 顯示資料
- 修改資料
- 呼叫特定函式

程式表面上看似正常運作，但其內部對於使用者輸入缺乏嚴格限制，使攻擊者可以透過輸入異常值影響程式執行流程。

本題的核心不涉及記憶體破壞，而是：

- 利用輸入改變程式邏輯
- 觸發非預期的程式分支
- 最終取得 flag

## 三、分析目標

- 分析程式的選單執行流程
- 找出輸入處理上的安全弱點
- 嘗試利用異常輸入影響程式邏輯
- 成功觸發隱藏功能
- 取得 flag

## 四、靜態分析

程式運作流程大致如下：

- 顯示選單
- 接收使用者輸入
- 根據輸入執行對應功能

但程式存在以下問題：

- 未限制輸入範圍
- 未驗證輸入合法性
- 未妥善處理異常輸入

這代表使用者輸入直接影響程式控制流程。

## 五、弱點判定

### 5.1 弱點描述

程式在處理選單輸入時，未對使用者輸入進行有效驗證，例如：

- 未檢查數值是否在合法範圍內
- 未限制輸入類型
- 未處理非法選項

導致攻擊者可以輸入：

- 超出範圍的數值
- 負數
- 非預期格式

進而影響程式邏輯。

### 5.2 弱點類型

- Improper Input Validation
- Logic Manipulation
- Unsafe Control Flow

### 5.3 風險說明

由於程式的執行流程依賴使用者輸入，當輸入未經驗證時，攻擊者可以：

- 控制程式執行分支
- 觸發未預期的功能
- 存取原本不應開放的邏輯

最終可能導致 flag 被洩漏。

## 六、攻擊思路

程式原始流程：

- 輸入合法選項 → 執行正常功能

攻擊目標：

- 輸入異常值 → 改變執行路徑 → 觸發隱藏功能

攻擊核心在於：

- 找出未被限制的輸入範圍
- 嘗試各種邊界值（boundary values）
- 利用邏輯漏洞進入非預期分支

## 七、利用過程

1. 觀察選單
2. 測試異常輸入（負數、極大數值、未定義選項）
3. 觸發異常分支
4. 取得 flag

## 八、利用結果

成功透過操控輸入改變程式邏輯，並觸發隱藏功能取得 flag。

## 九、完整 exploit 概念

- 輸入異常值
- → 改變程式邏輯
- → 觸發隱藏功能
- → 取得 flag

## 十、成因分析

1. 缺乏輸入驗證
2. 過度信任使用者輸入
3. 控制流程設計不完整

## 十一、防禦建議

- 驗證輸入範圍
- 加入 default case
- 限制輸入格式
- 最小信任原則

## 十二、結論

本題展示了一個典型的邏輯漏洞（Logic Vulnerability）。

此案例說明，在軟體開發中，即使沒有低階漏洞，邏輯設計錯誤本身也可能造成安全風險。
