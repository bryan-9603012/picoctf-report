## 二、題目概述

本題透過 netcat 提供互動式服務，要求使用者根據題目敘述輸入指定格式的內容。
在本關中，服務端提示如下：

Send me ASCII DECIMAL 101, 101, 101, side-by-side, no space.

題目表面上看似只是將數字直接拼接，但實際上考察的是對 ASCII 十進位編碼 的理解。使用者必須先將十進位數值轉換為對應字元，再依題意將字元連續輸出。

## 三、分析目標

本次分析的主要目標如下：

1. 判讀題目要求的資料型態
2. 確認 ASCII DECIMAL 101 的實際意義
3. 將數值正確轉換為 ASCII 字元
4. 依題目格式送出答案
5. 說明本題考點與常見誤區

## 四、題目分析

題目要求輸入：

ASCII DECIMAL 101, 101, 101

在 ASCII 編碼中：

十進位 101 對應字元為 e

因此三個 101 代表三個字元：

e e e

又因題目要求：

side-by-side, no space

所以正確輸出應為：

eee
## 五、錯誤判斷與修正

### 5.1 常見錯誤

容易出現的錯誤是直接將數字拼接為：

101101101

這種做法是把題目中的十進位 ASCII 值視為原始輸入內容，而不是先轉成字元，因此會被服務端判定為錯誤。

### 5.2 正確理解

本題不是要送：

數字字串 101101101

而是要送：

ASCII 十進位 101 對應的字元 e
重複三次後形成 eee

## 八、解題核心觀念

本題重點在於理解：

- ASCII DECIMAL 表示的是十進位 ASCII 碼
- 必須先將數值轉換成字元
- side-by-side, no space 表示最終輸出不得包含空白或分隔符

## 九、成因分析

此題的主要設計目的，是測試使用者是否真正理解基本字元編碼，而非僅依表面格式輸入數字。
若對 ASCII 十進位、字元映射關係不熟悉，很容易誤把題目理解為單純字串拼接題。

## 十、防禦建議／學習建議

對於類似題型，建議建立以下判讀流程：

先辨識資料型態（ASCII / HEX / BYTE / DECIMAL）
確認題目要求的是「字元」還是「原始位元組」
再決定應輸出可見字串還是 raw bytes
## 十一、結論

bytemancy 0 的核心在於 ASCII 十進位到字元的轉換。
題目要求的 101,101,101 並非直接拼接為數字，而是應先轉換成字元 e，最終輸出 eee。本題屬於基礎但重要的編碼理解題。
